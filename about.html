How I approach development

Semantic structure first

Before I think about design or interactivity, I think about structure. A page built on solid semantic HTML communicates meaning to browsers, screen readers, search engines, and future developers — not just the person currently looking at it. That foundation matters.

I use elements like header, nav, main, section, article, aside, and footer because they carry meaning. Generic div containers don't. That's a small discipline that compounds over time into code that is easier to maintain and easier for everyone to use.

Accessibility is not optional

Web accessibility isn't a feature to add at the end — it's a standard that shapes how I build from the beginning. Every project I deliver is tested with WAVE and reviewed with screen reader tooling. I implement ARIA attributes, logical heading hierarchies, visible focus states, and color contrast that meets WCAG AA standards.

Testing my own sites in high-contrast mode and with VoiceOver has taught me more about accessibility than any documentation. It changed how I write markup at a fundamental level.

Details are the product

In my experience, the difference between a site that feels trustworthy and one that feels rough is almost always in the details — heading hierarchy, consistent spacing, labels that match their inputs, error states that are actually helpful. Users don't consciously notice these things when they're right. They notice immediately when they're wrong.
